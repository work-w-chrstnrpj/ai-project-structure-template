#Implementation Brief: Lean AI Project Structure Template Refactor

Use this file as the task brief for OpenCode when updating the repository:
https://github.com/work-w-chrstnrpj/ai-project-structure-template

## 0. Mission

Refactor the repo so it follows the project goal:

- Project-agnostic, framework-agnostic, language-agnostic AI project template.
- Works across Codex, Claude Code, Antigravity, GitHub Copilot, OpenCode, Aider, Cursor, OpenHands, Cline, Kilo Code, and Windsurf.
- Quality over quantity.
- Minimal number of agents, skills, tools, workflows, and adapters that produce maximum practical value.
- Token-efficient and context-efficient.
- `.agents/` is the canonical source tree.
- Tool-specific folders are generated adapters, not hand-maintained source of truth.

Current issue to fix:

The repo currently has the right core idea, but `.agents/skills/` has expanded into too many granular capability folders. Many of those folders are not true reusable skills. They are domain capabilities that belong inside the owning agent role. This causes skill bloat, adapter bloat, and more context noise.

The desired end state:

- Keep the 8 core engineering agents.
- Add 2 overlays: `mentor` and `coach`.
- Keep only standalone skills that are true reusable executable workflows.
- Move non-standalone capability guidance into the appropriate role files as embedded playbooks.
- Update the adapter generation script so tool-specific folders can be created/synced from canonical `.agents/` files.
- Preserve the repo's project-agnostic nature.

Do not add project-specific implementation assumptions. This is a reusable template, not an app repo.

## 1. Core architectural decision

Use this model everywhere in the repo:

```text
Canonical source layer:
- AGENTS.md
- .agents/roles/
- .agents/skills/
- .agents/overlays/
- .agents/workflows/
- .agents/tools/

Generated adapter layer:
- .claude/
- .codex/
- .cursor/
- .github/
- .opencode/
- .clinerules/
- .kilo/
- .windsurf/
- Aider adapter files such as .aider.conf.yml and CONVENTIONS.md
```

Rule:

```text
Humans edit canonical files under .agents/ and AGENTS.md.
Scripts generate or sync tool-specific adapters.
Do not hand-edit generated adapters unless explicitly instructed.
```

Important distinction:

```text
A standalone skill = reusable workflow with actual procedure and output.
An agent role = domain ownership, responsibilities, handoffs, and embedded capability playbooks.
An overlay = response style or thinking posture, not a work owner.
A workflow = multi-role process across agents and skills.
A tool doc = external tool usage policy, context policy, or command policy.
```

### 3.3 Criteria for future standalone skills

Add this rule to `.agents/skills/README.md`, `.agents/README.md`, and `AGENTS.md`:

A standalone skill is allowed only if it satisfies all of these:

1. It defines a repeatable workflow, not just an area of expertise.
2. It can be reused across projects.
3. It is useful across more than one role/tool, or it is important enough to standardize.
4. It has clear inputs, workflow steps, boundaries, and output expectations.
5. It helps context efficiency rather than adding routing noise.

Do not create standalone skills for every agent capability. Put domain-specific checklists and procedures inside the owning role file.

## 4. Move non-generic skills into agents

Each role file should replace the old `## Allowed Skills` section with:

```md
## Reusable Skills

- `ask-repo-question` when the task is explanation-only or repo Q&A.
- `investigate-issue` when diagnosing a bug, incident, failed test, or unexpected behavior.
- `plan-code-change` when planning a feature, bugfix, refactor, or chore without editing.
- `execute-code-change` when implementing, fixing, modifying, updating, refactoring, or cleaning code.
- `code-review` when reviewing changed code.
- `use-context-tools` when selecting efficient context paths and discovery tools.
- `write-automated-test-cases` when executable test coverage is needed.

- `execute-manual-test-cases` when manual QA execution is requested.
- `update-docs` when docs must change after behavior, API, database, architecture, deployment, testing, or AI workflow changes.
- `release-check` when preparing for merge or release.
```

Only list skills that actually make sense for that role. Do not mechanically include every skill in every role if it would be noisy.

```md

```

```md
### Capability: <capability name>

Use when:
- <when this capability applies>

2. <identify affected boundaries>
3. <apply role-specific rules>
4. <coordinate handoffs when needed>
5. <define verification/docs requirements>

Guardrails:
- <what this role must not do>
- <what requires escalation>

Output:
- <expected output 1>
- <expected output 2>
```

# Overlays

Overlays change response style, explanation depth, or feedback posture. They do not own implementation, architecture, QA, security, deployment, or documentation decisions.

Examples:

```md
# Mentor Overlay

Mentor is an explanation and teaching overlay. It changes how the response is explained, but it does not replace the owning engineering role.

## Use When

Use Mentor when the user wants to learn, understand, clarify, or receive a guide.

## Invocation Examples

- `Mentor: Explain this architecture.`
- `Mentor ELI5: Explain dependency injection.`
- `Mentor ELI12: Explain how the adapter generator works.`
- `Backend & Database Engineer + Mentor: Explain this API design.`
- `SQA Engineer + Mentor: Teach me how to write test cases for this flow.`

## Behavior

- Be concise without omitting important details.
- Do not assume the user already knows the topic.
- Break ideas into digestible parts.
- Use simple language first, then introduce correct technical terms.
- Define important terms, acronyms, slang, or vocabulary.
- Use relatable analogies when helpful.
- Paint a mental picture of how the concept works.
- Use the requested ELI level when provided.
- Only use ELI-style explanation when the user asks for ELI, clarification, or teaching.

## Tone

- Supportive.
- Patient.
- Optimistic.
- Clear.
- Respectful.
- Knowledgeable without being arrogant.

## Output Shape

Prefer:

1. Simple explanation.
2. Key terms.
3. Example or analogy.
4. Practical takeaway.

## Boundaries

- Mentor does not own implementation, architecture, QA, security, deployment, or delivery decisions.
- Combine Mentor with the appropriate role when domain ownership is needed.
```

Coach is a critical thinking and decision-quality overlay. It changes how feedback is delivered, but it does not replace the owning engineering role.

- Direct.
- Critical.
- Pragmatic.
- Constructive.
- High standards.
- No sugar-coating.
- Respectful but not overly gentle.

## Output Shape

Prefer:

1. What is right.

3. What is missing.
4. Better alternative.

## Boundaries

- Coach does not own implementation, architecture, QA, security, deployment, or delivery decisions.

### 6.4 `execute-code-change`

Rewrite this skill to support both plan-first and direct execution.

Skill frontmatter:

```md
---
name: execute-code-change
description: Implement a code change end to end. Use when the user asks to execute, implement, fix, modify, update, refactor, clean up, or add behavior in the repo. Can follow an existing plan or perform a small direct implementation safely.
---
```

Required content:

```md
# Execute Code Change

## Purpose

Implement a safe, source-grounded code change with verification and documentation updates when needed.

## Execution Modes

### Mode A: Plan-first execution
Use when:
- The user provides a plan.
- `plan-code-change` was already run.
- A ticket or implementation plan exists.

Procedure:
1. Read the plan.
2. Validate the plan against current files.
3. Identify any stale assumptions or missing files.
4. Execute the smallest safe slice.
5. Add/update tests when behavior changes.
6. Update docs when behavior, API, database, architecture, deployment, project structure, testing, or AI workflow changes.
7. Run the smallest relevant verification.
8. Report deviations from the plan.

### Mode B: Direct execution
Use when:
- The user directly asks to implement, fix, update, refactor, or clean up.
- No separate plan exists.
- The change is small or medium-risk.

Procedure:
1. Do a brief internal planning pass.
2. Read `AGENTS.md`.
3. Use `.ai/context-routing.md` if present.
4. Use maps/indexes/tools only for discovery.
5. Read exact source files, nearby tests, contracts, and docs before editing.
6. Confirm the smallest safe implementation that follows existing patterns.
7. Edit narrowly.
8. Add/update tests when behavior changes.
9. Update docs when needed.
10. Run the smallest relevant verification command.
11. Summarize files changed, behavior changed, verification, and remaining risks.

## Escalate to `plan-code-change` first when

- The change spans multiple modules.
- Requirements are unclear.
- Public API behavior changes.
- Auth/security behavior changes.
- Data migrations are involved.
- Deployment/runtime behavior changes.
- The blast radius is unknown.
- The user explicitly asks to plan first.

## Boundaries

- Do not bypass project guardrails in `AGENTS.md`.
- Do not expose secrets or sensitive data.
- Do not run destructive commands, migrations, deployments, force pushes, resets, or cleanups without explicit approval.
- Do not claim verification passed unless it actually ran.
- Do not edit from generated context alone.

## Required Output

- Execution mode used.
- Relevant files/docs inspected.
- Files changed.
- Behavior changed or structure changed.
- Tests/verification run, including command and result.
- Docs updated or skipped with reason.
- Deviations from plan, if any.
- Remaining risks, assumptions, and handoffs.
```

### 7.2 Add target-aware sync script

Create or update `scripts/sync-ai-adapters.ps1`.

It should support:

```powershell
param(
  [ValidateSet(
    "all",
    "claude",
    "codex",
    "cursor",
    "copilot",
    "opencode",
    "aider",
    "cline",
    "kilo",
    "windsurf",
    "antigravity",
    "openhands"
  )]
  [string[]]$Target = @("all"),

  [switch]$Clean
)
```

If `-Target all` is passed, generate all supported adapters.
