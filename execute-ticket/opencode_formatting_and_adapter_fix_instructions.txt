## Context From Prior Review

The repo has improved substantially and now mostly meets the intended structure, but it is not fully ready because formatting likely breaks script/tool compatibility.

Examples of current problems to fix:

1. `AGENTS.md` appears compressed into only a small number of raw lines. Headings, lists, and paragraphs are not reliably separated by real line breaks.

2. Canonical role files such as `.agents/roles/backend-database-engineer.md` appear to contain metadata like `# Agent`, `## Identity`, `- **Name:**`, `- **Role:**`, and `- **Description:**` on the same line. This is bad for readability and may break scripts that expect metadata at line starts.

3. Skill files such as `.agents/skills/plan-code-change/SKILL.md` currently appear to start like this:

   ```md
   --- name: plan-code-change description: Plan a code change without editing files. ... --- # Plan Code Change
   ```

   This is not valid YAML frontmatter. It must be converted to a proper multi-line frontmatter block.

4. Generated adapter files such as `.opencode/agents/coach.md` currently appear to have compressed frontmatter like this:

   ```md
   --- name: coach description: 'Overlay: coach. Changes response style, not work ownership.' mode: primary permission: edit: deny bash: ask ---
   ```

   This should be proper multi-line YAML frontmatter.

5. `scripts/sync-ai-adapters.ps1` appears compressed into a few long lines. PowerShell may still parse some of it, but it is hard to review and may contain accidental embedded line breaks inside strings. Reformat it into normal, readable, maintainable PowerShell.

6. `scripts/sync-ai-adapters.ps1` currently expects role metadata to match regexes such as start-of-line `- **Name:**`. That means canonical role metadata must appear at the start of actual lines.

7. `scripts/sync-ai-adapters.ps1` expects skill frontmatter to match a real multiline `---` YAML block. Therefore every `SKILL.md` must use proper frontmatter.

## Desired Canonical Formatting Standards

### Markdown Standard

Every Markdown file should use normal line breaks.

Use:

```md
# Heading

Intro paragraph.

## Section

- Bullet one.
- Bullet two.

### Subsection

1. Step one.
2. Step two.
```

Do not use compressed one-line markdown like:

```md
# Heading Intro paragraph. ## Section - Bullet one. - Bullet two.
```

### Role Metadata Format

Every canonical role file under `.agents/roles/*.md` must begin with a parseable identity section like this:

```md
# Agent: Backend & Database Engineer

## Identity

- **Name:** backend-database-engineer
- **Role:** Backend & Database Engineer
- **Description:** Owns backend services, APIs, business logic, data models, integrations, batch work, queues, and data ingestion.
```

Requirements:

- `- **Name:**` must start at the beginning of a real line.
- `- **Role:**` must start at the beginning of a real line.
- `- **Description:**` must start at the beginning of a real line.
- Keep `Name` values lowercase kebab-case.
- Keep role names human-readable.
- Keep descriptions concise but useful.

### Overlay Format

Every overlay under `.agents/overlays/*.md` should use a clear Markdown structure.

Example:

```md
# Coach Overlay

Coach is a critical thinking and decision-quality overlay. It changes how feedback is delivered, but it does not replace the owning engineering role.

## Use When

Use Coach when the user wants brainstorming, critique, tradeoff analysis, decision review, or direct feedback.

## Invocation Examples

- `Coach: Critique this idea.`
- `System Architect + Coach: Challenge this architecture proposal.`

## Behavior

- Do not agree by default.
- Identify what is strong, weak, missing, risky, duplicated, or overcomplicated.
```

### Skill Frontmatter Format

Every `.agents/skills/*/SKILL.md` file must begin with valid multiline YAML frontmatter.

Use this shape exactly:

```md
---
name: plan-code-change
description: Plan a code change without editing files. Use for feature work, bug fixes, refactors, chores, cleanup, dependency/config updates, or implementation scoping before code changes.
---

# Plan Code Change

## Purpose

Create a source-grounded implementation plan without editing files.
```

Requirements:

- Opening `---` on its own line.
- `name:` on its own line.
- `description:` on its own line.
- Closing `---` on its own line.
- Blank line after closing frontmatter.
- `# Skill Name` heading after frontmatter.
- No compressed frontmatter.
- No JSON-style frontmatter.

```md

- **Role:** value
- **Description:** value
```
